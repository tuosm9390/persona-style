# 📝 프로젝트 개선 사항 이행 계획서: PersonaStyle

본 계획서는 PersonaStyle 프로젝트의 안정성, 성능, 및 신뢰성을 한 단계 높이기 위해 도출된 4가지 핵심 개선 사항의 상세 이행 방안을 담고 있습니다.

## 1. 목표 (Goal)
- **Stability**: 외부 모니터링 도구 통합 및 에러 처리 체계화를 통한 운영 안정성 확보.
- **Performance**: 대규모 리소스 생성(PDF)의 서버 이관을 통한 클라이언트 성능 최적화.
- **Reliability**: E2E 테스트 도입을 통한 핵심 비즈니스 로직의 완벽한 검증.

## 2. 단계별 작업 계획 (Phased Roadmap)

### Phase 1: 관측 가능성 및 에러 처리 고도화 (안정성 강화)
- **Task 1.1: Sentry 통합 및 로거 확장**
- `src/lib/logger.ts`를 Sentry SDK와 연동하여 에러 발생 시 실시간 알림 체계 구축.
- 운영 환경에서의 로그 레벨별 필터링 및 추적 ID(requestId) 연동 강화.
- **Task 1.2: Gemini API 에러 처리 리팩토링**
- `src/app/api/analyze/route.ts`의 문자열 매칭 기반 에러 처리 로직을 구글 공식 에러 코드 기반의 매핑 테이블 방식으로 전환.
- `ERROR_CODE_MAP` 상수를 정의하여 유지보수성 향상.

### Phase 2: 서버 측 PDF 생성 전환 (성능 최적화)
- **Task 2.1: Supabase Edge Functions 환경 구축**
- PDF 생성을 담당할 별도의 Edge Function 프로젝트 설정 및 라이브러리 선정.
- **Task 2.2: PDF 생성 로직 이관**
- `src/lib/pdf.tsx`의 로직을 서버 측으로 마이그레이션.
- 분석 결과 데이터를 바탕으로 서버에서 PDF를 생성하고 Supabase Storage에 자동 업로드하는 파이프라인 구현.
- **Task 2.3: UI/UX 업데이트**
- 클라이언트에서 직접 생성하던 방식을 서버 생성 요청 후 Storage URL을 다운로드하는 방식으로 변경.
- 생성 진행 상태를 나타내는 단계별 프로그레스 UI 적용.

### Phase 3: E2E 테스트 도입 (신뢰성 확보)
- **Task 3.1: Playwright 환경 설정**
- 프로젝트 내 Playwright 설치 및 브라우저 테스트 환경 구성.
- **Task 3.2: 핵심 시나리오 테스트 작성**
- [분석 요청 -> 결제 프로세스 -> 프리미엄 리포트 확인]으로 이어지는 'Golden Path' 전체를 검증하는 시나리오 작성.
- 다국어 전환 및 모바일 반응형 레이아웃 정상 동작 확인.

## 3. 리스크 및 대응 방안 (Risk & Mitigation)
- **PDF 폰트 호환성**: 서버 측 PDF 생성 시 한글 폰트 깨짐 현상이 발생할 수 있으므로, 나눔고딕 등 웹 폰트의 서버 임베딩 검증 필수.
- **Edge Function 실행 제한**: 대용량 PDF 생성 시 실행 시간 초과 가능성이 있으므로, 비동기 큐 방식 또는 람다 함수 호출 검토.

## 4. 검증 및 완료 조건 (Verification)
- [ ] Sentry 대시보드에 테스트 에러가 정상적으로 집계되는가?
- [ ] API 응답 메시지가 변경되어도 에러 처리 로직이 견고하게 동작하는가?
- [ ] 서버에서 생성된 PDF 파일이 Storage에 보안 정책(RLS)을 준수하며 저장되는가?
- [ ] Playwright E2E 테스트가 모든 브라우저 환경에서 Pass 하는가?

## 5. 결론
위 계획에 따라 작업을 진행함으로써 PersonaStyle은 단순한 기능 구현을 넘어 프로덕션 급 서비스로서의 견고한 기반을 갖추게 될 것입니다. 각 단계 완료 후 개별 작업 보고서를 통해 진행 상황을 공유하겠습니다.
